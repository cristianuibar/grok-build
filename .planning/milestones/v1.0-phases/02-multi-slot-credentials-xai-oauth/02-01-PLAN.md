## Phase Goal

**As a** bum user, **I want to** store xAI credentials in a multi-provider auth file under bum home, **so that** a later Codex slot cannot be wiped by xAI login and agent turns still authenticate on the xAI path.

<objective>
Deliver the multi-slot on-disk `auth.json` core: `AuthDocument` schema, legacy flat → `providers.xai` read migration, **dual lock-scoped mutation APIs** (acquiring vs caller-held guard) that emit nested form with `version: 1`, rewire existing lock-holding manager/enrichment writers to the guard-held API, concurrency-safe sibling preservation, and fail-closed unsupported-version handling (per D-01, D-02, D-03, D-04, D-05, D-10, D-11, D-14).

Addresses review concerns: unlocked merge RMW; concurrent clobber; **lock reentrancy / self-deadlock when write re-acquires under held AuthFileLock** (cycle 2); unsupported schema version treated as corrupt-empty recovery; accidental public widening of write surface.

Purpose: Dual-OAuth-safe storage is the foundation for AUTH-01 under bum; AuthManager stays storage-agnostic via the xAI-slot adapter (D-12).
Output: Working multi-slot storage layer with dual mutation APIs + manager/enrichment rewired + automated isolation/concurrency/no-deadlock tests; no Codex login (D-15), no stock-home import (D-13).
</objective>
