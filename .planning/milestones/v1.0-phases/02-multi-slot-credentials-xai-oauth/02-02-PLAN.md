## Phase Goal

**As a** bum user, **I want to** clear an xAI API key, log out of xAI, or recover xAI auth without wiping a reserved sibling provider slot, **so that** multi-provider credentials stay isolated under one auth.json.

<objective>
Make API-key helpers, scope-removal empty-delete, and **all whole-file purge sites** (including `try_devbox_recovery`) multi-provider-aware and **lock-scoped via Plan 01 dual mutation APIs** (per D-05, D-09, D-11). After Plan 01 dual lock-scoped merge-safe writes, empty-xAI paths must not `remove_file` when `providers.codex` (or any sibling) still holds scopes (AUTH-01 isolation / Phase 5 readiness).

Addresses review concerns: try_devbox_recovery whole-file purge; unlocked store_api_key/clear_api_key RMW; incomplete prune/delete policy; FileDeleted without successful delete; **cycle 2: keep already-locked manager/enrichment on guard-held path when prune/devbox touch those writers**.

Purpose: Prevent the classic empty-map delete pitfall that would log out Codex once Phase 5 fills the slot.
Output: Shared locked prune-or-delete helper (acquiring + with-lock); `store_api_key` / `clear_api_key` / `write_scope_removal` / `try_devbox_recovery` isolation tests green; no reentrancy regression.
</objective>
