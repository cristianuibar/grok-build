<objective>
Make the mid-session model switch itself sticky and clamp-aware, and surface the clamp to the user — using
a genuinely separate raw-preference state, not a reused resolved/effective field. Cross-AI review
(`11-REVIEWS.md`, HIGH findings 2-4) found the original design's `handle.reasoning_effort =
applied_effort` mechanism let an implicit catalog default become a sticky "user preference" (Sol's `low`
default would incorrectly stick when switching on to Terra, which should get its own `medium` default),
because `SessionHandle.reasoning_effort` is seeded from resolved sampling config at session creation, not
from an explicit-choice-only store. This revision introduces a dedicated `SessionHandle
.reasoning_effort_preference: Option<ReasoningEffort>` field, written ONLY from explicit user action
(`effort_override` on this switch, or carried forward from the field's own prior value — never from a
catalog default that merely flowed through `prepared.sampler_config.reasoning_effort`). It also fixes a
wire-contract bug (HIGH#4): the response's resolved effort serializes to JSON `null` when the target model
has an empty supported list (a deliberate "no effort applies" signal), which the original pager-side parse
could not distinguish from "key absent" (legacy-server fallback) — reintroducing the exact stale-value bug
this phase exists to fix. The session's chosen RAW effort preference now survives every switch — including
a Grok->Codex->Grok round-trip returning the exact original value, and a no-preference Sol->Terra switch
correctly picking up Terra's OWN catalog default instead of Sol's. The switch handler never silently drops
an unsupported override (RESEARCH.md §B5 finding 3/4) — it always stores the raw preference and lets
`11-01-PLAN.md`'s choke point clamp it per-request. A one-line, non-blocking TUI notice (locked copy from
`11-UI-SPEC.md`) tells the user when a switch produced an effective effort different from what they had,
built from the exact same catalog view the shell used for the clamp decision (fixing review finding M5's
atomicity gap), and a clamped switch never contaminates the user's persisted global default effort (fixing
review finding M4).

Purpose: closes phase goal criteria 2 (soft-clamp, never hard-fail) and criterion 4 (UAT notes no longer
list effort as a blocker). Depends on `11-01-PLAN.md`'s `clamp_reasoning_effort` helper and
`reasoning_effort_supported` field — this plan calls the same shared function a second time (mirrored,
display-only) at switch time, per CONTEXT's "shared function all callers route through."
Output: a dedicated sticky per-session RAW effort-preference field (distinct from the resolved/effective
field), a clamp-aware switch handler that never conflates the two, an ACP response meta carrying the
resolved effort (tri-state null-vs-absent safe) + clamped flag + the exact supported list used, a pure
response-parsing helper the pager can unit-test directly, a TUI notice built from that same response data,
clamp-safe persistence, behavior + TUI test tiers (redirected to assert via response metadata and
dispatched `TaskResult` fields — never `SessionHandle` internals, per review finding L2), and UAT-notes
closure.

Cycle-2 review found the interactive mid-session-switch path this revision targeted was correctly fixed,
but three more findings surfaced on paths the original revision did not reach: (1) session resume/restore
(`agent_ops.rs`/`acp_agent.rs::load_session`) reintroduces the exact raw-preference/catalog-default
conflation via a different door — a fresh session's bare catalog-default effort gets persisted and, on
resume, replayed through `model_switch::apply` as if it were an explicit user override (review finding
HIGH#2/NEW-B); (2) deleting the `model_supports_reasoning_effort` gate universally (rather than for Codex
targets only) lets a stored Codex effort preference reach a wire request for a Grok model variant that
hard-rejects the `reasoningEffort` parameter with a 400, regressing CONTEXT's locked "Grok/xAI path
untouched" decision (review finding HIGH#3/NEW-C); (3) the switch handler's `supported` list for both the
clamp decision and the notice was derived from a raw `model_reasoning_efforts(model_id)` map-key lookup
rather than the same alias/routing-slug-resolved entry `sampling_config_for_model` uses, reopening the
atomicity gap M5 was meant to close for a model addressed by an alias (review finding A2). This revision
adds a dedicated Task 2 for the resume/restore fix, restores the Grok-path gate and fixes the alias
resolution in Task 1, and adds regression tests for all three plus the two remaining A3 compile-ripple
sites in the renumbered Task 3.
</objective>

<assumption_delta_decision>
**Noun now primary:** reasoning effort identity is now explicitly THREE distinct values at every switch:
(1) `SessionHandle.reasoning_effort_preference` — the raw, sticky, per-session preference, written only by
explicit user action; (2) the transient request candidate stamped into `prepared.sampler_config
.reasoning_effort` for this switch (the raw preference if one exists, else the target model's catalog
default — never written back into (1)); (3) `SessionHandle.reasoning_effort` — the resolved/effective,
post-clamp DISPLAY value used for broadcast, status, and the "Switched to..." notice (unchanged role from
before this phase, just now correctly reflects the post-clamp value).
**Decision:** no-change.
**Rationale:** no required->optional field transition occurred — the assumption-delta detector fired on
the word "optional" in the ROADMAP line about the explicitly skipped ultra->max ladder UI (a skipped
feature, not a field transition). The additions this plan makes are additive-only: `reasoning_effort_
preference` is a new `Option<ReasoningEffort>` field on `SessionHandle` (seeded `None`, matching the type
and optionality of its sibling `reasoning_effort` field — no existing field's type or optionality
changes), `reasoning_effort_clamped`/`reasoning_effort_supported` are new optional ACP response-meta keys
(absent/empty = no clamp / no menu), and `TaskResult::SwitchModelComplete` gains two new fields. Cross-AI
review (HIGH findings 2/3) found the ORIGINAL plan's mechanism — reusing the existing
`SessionHandle.reasoning_effort` field as if it already held "explicit choice only" — was itself the bug:
that field is seeded from resolved sampling config at session creation (`spawn.rs:1613`), so it cannot
distinguish user choice from a catalog default. This revision's dedicated new field is the actual fix, not
a reinterpretation of an existing field's meaning. Cycle-2 review (finding NEW-B) found the SAME
distinction was missing one hop further out: the on-disk persistence layer (`PersistenceMsg::CurrentModel`)
also only carried the resolved/effective effort, with no separate raw-preference field, so a value of
unknown provenance could round-trip through a process restart and re-emerge as a false "explicit choice."
The new `reasoning_effort_preference` persisted field this revision's Task 2 adds is additive-only in the
same sense as `SessionHandle`'s field — no existing persisted field's type or optionality changes.
</assumption_delta_decision>
