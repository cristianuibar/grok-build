**GROK_AUTH_PATH policy (CONTEXT D-08 + review HIGH + cycle 2 symlink HIGH + orchestrator discretion):** Prefer resolved product-home `auth.json` (`{grok_home}/auth.json` where grok_home is BUM_HOME / `~/.bum`) for **all login/refresh writes**.

**Acceptance of override (cycle 2 — mandatory, no lexical-only escape):** Accept override **only if** one of:
1. **Exact match:** `GROK_AUTH_PATH` resolves to the same path as `{product_home}/auth.json` (string/Path equality after normalizing `.` components as needed), **or**
2. **Canonical containment:** canonicalize the override’s **existing parent directory** with **`dunce::canonicalize`** (project convention — never `std::fs::canonicalize` / `Path::canonicalize` per clippy.toml) and verify the canonical parent is under the canonical product home; the file leaf must be `auth.json`.

**Do not** treat bare `path.starts_with(product_home)` lexical checks as sufficient — an in-home symlink can escape to `~/.grok` or elsewhere, and `open_secure_file` in `xai-grok-shell-base` follows symlinks on open.

If the override fails acceptance, **ignore it for writes** and persist to product-home `auth.json`. Production must not write credentials into `~/.grok/auth.json` or through escape symlinks. Read of a foreign override path is not required for production this phase; migration tests use temp homes only.

**Tests required:** (a) foreign stock-like path does not receive writes; (b) Unix: symlink under product home pointing outside does not receive credential writes (target and link path both must remain free of credentials; product-home auth.json receives the write).
