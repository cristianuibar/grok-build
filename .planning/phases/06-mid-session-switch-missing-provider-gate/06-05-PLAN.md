## Session harness contract (non-negotiable)

Do **not** treat `provider_routing::switch_changes_next_sample_route` as proof of history preservation or mid-turn non-cancel — it only composes routing transforms without MvpAgent, chat history, or in-flight turns (review finding).

**Required harness** (extend Plan 01 `tests/model_switch_gate.rs` or co-located helpers):

1. Real session path capable of `model_switch::apply` / ACP `set_session_model` with dual-slot fixture auth and mixed catalog.
2. **History:** snapshot conversation/message count (or chat-state length via session query) **before** successful cross-provider switch; after apply assert same length and that prior user/assistant content still present (or stable history ids).
3. **Mid-turn non-cancel:** start or mark an in-flight turn if the harness supports it; call apply; assert session does not issue cancel-turn / abort for the in-flight turn (observe cancel channel absence, turn_count still active, or documented session command that cancel was not sent). If full concurrent turn is too heavy, use the lightest signal the real apply path exposes that cancel is not invoked (e.g. spy/wrapper on cancel entry points, or assert apply success path code path does not call cancel — plus a test that session actor remains accepting prompts).
4. **Next-turn provider:** after successful switch, assert prepared sampling / next sample credentials / base_url match the **new** provider (reuse Phase 4 route assertions at session level, not only pure transform unit test).
5. All free-switch / history / mid-turn tests use unique `p6_` prefixes.

`provider_routing.rs` may still host pure routing regression `switch_changes_next_sample_route` (already green) — optional dual_login pure helpers only if they do not claim session history proofs.
