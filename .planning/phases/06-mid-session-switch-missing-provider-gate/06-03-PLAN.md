## Discretion lock (Login now depth)

1. **xAI Login now:** Reuse existing mid-session login / Authenticate effect path when provider is xai (dispatch_login / Effect::Authenticate patterns in auth.rs).
2. **Codex Login now:** If no thin Codex TUI authenticate effect exists, close QuestionView, set deferred_model_switch with required_provider=codex, push system/scrollback or toast restating `bum login --provider codex`, and start **provider-status refresh wait** (below). Do **not** call xAI interactive login for a Codex-missing gate.
3. Never apply the pending model until the required provider is usable **and** shell switch gate would pass (shell still authoritative on retry).

## External CLI → deferred apply (contract)

`handle_auth_complete` alone is insufficient for CLI-primary Codex (review HIGH). Required path:

1. **Producer:** Reuse Plan 04 refresh of dual-slot usable flags (AuthStatusReport / AuthMeta providers / Effect::RefreshProviderAuthStatus). Pure booleans only — no tokens in AppView.
2. **Pinned scheduling contract (cycle 2 — no open choice):**
   - After CLI-primary Login now: set `awaiting_provider_login: Some(provider)` (or equivalent on AgentSession/AppView) and arm a **bounded poll** with a **generation counter** (u64 bumped on each arm/clear). Emit Effect::RefreshProviderAuthStatus on arm, then on each poll tick while deferred still pending and generation matches — max N attempts (e.g. 30) or max window (e.g. ~2 minutes); then stop polling (user can still FocusGained).
   - On `Event::FocusGained` while deferred is pending with required_provider: **always** emit Effect::RefreshProviderAuthStatus (Plan 04 also refreshes for badges unconditionally — this plan may share that emission; if Plan 04 already emits on every FocusGained, deferred consumer runs on TaskResult only; still test that FocusGained path results in a refresh attempt while awaiting).
   - On deferred take/clear / Keep current / abandon: bump generation (or clear awaiting flag) so in-flight poll TaskResults with stale generation are no-ops (no SwitchModel, no spurious apply).
3. **Consumer:** `try_apply_deferred_model_switch_if_ready(app)`:
   - If deferred is None → no-op
   - If deferred.required_provider is Some(p) and !cache.usable(p) → keep deferred, no SwitchModel
   - If required provider usable (or required_provider is None for legacy CLI `-m` deferrals) → take deferred, emit Effect::SwitchModel with model_switch_pending **and persist_default from the deferred struct**
4. Unrelated successful auth for the other provider must **not** apply a codex-targeted deferred while codex still unusable.

## DeferredModelSwitch shape (cycle 2)

```text
DeferredModelSwitch {
  model_id,
  effort: Option<ReasoningEffort>,
  required_provider: Option<ProviderWireId>, // Some from MissingProviderLogin; None for CLI -m
  persist_default: bool, // true when stashed from set_default_model / settings path (Plan 02)
}
```

Session-created apply and post-login retry must re-issue SwitchModel with the same persist_default so settings default persist/toast intent is not lost (cycle 2 MEDIUM).
